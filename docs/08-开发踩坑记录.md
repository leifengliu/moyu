# 开发踩坑记录

## 1. 部署后前端接口全部 302 + 后端报 Connection reset by peer

**现象**：本地开发一切正常，部署到服务器后前端接口返回 302，后端日志反复打印 `AsyncRequestNotUsableException: Connection reset by peer`。

**根因**：微信小程序生产环境**强制要求 HTTPS**，而 `app.js` 中 `baseUrl` 配的是 `http://`。请求被拦截/重定向到 `https://`，客户端收到 302 后断开当前连接去追重定向，此时服务端异步写响应发现连接已被 reset，抛出异常。

**解决**：

1. `app.js` 的 `baseUrl` 使用 HTTPS：
   ```javascript
   baseUrl: 'https://你的域名'  // 不加端口，Nginx/网关代理
   ```
2. 服务器必须配置 SSL 证书（免费申请：阿里云 DigiCert / 腾讯云 / Let's Encrypt）
3. [微信公众平台](https://mp.weixin.qq.com) → 开发管理 → 开发设置 → 服务器域名，将域名加入 **request 合法域名**
4. 如果不用 Nginx，可直接在 Spring Boot 中配置 SSL：
   ```yaml
   server:
     port: 8443
     ssl:
       key-store: classpath:keystore.p12
       key-store-password: 密码
       key-store-type: PKCS12
   ```
5. `GlobalExceptionHandler` 中可忽略此异常减少日志噪音：
   ```java
   @ExceptionHandler(AsyncRequestNotUsableException.class)
   public void handleAsyncRequestNotUsable() {
       // 客户端主动断开连接（如 302 重定向），无需处理
   }
   ```

## 2. 微信小程序 input 组件样式兼容性

### 2.1 文字被裁切（只显示下半部分）

**现象**：输入框内 placeholder 和文字只显示下半部分，上半部分被盖住。

**根因**：微信小程序的 `<input>` 组件有原生最小高度（约 24-26px），只通过 `padding` 撑高元素时，内容区被 `box-sizing: border-box` 挤压到几乎为零，文字就会溢出被裁剪。而普通 `<view>` 元素没有这个原生高度限制。

**错误示例**：
```html
<input style="height: rpx;" class="field-input-box" ... />  <!-- style值无效 -->
```
```css
.field-input-box {
  padding: 24rpx;
  box-sizing: border-box;
  /* 缺了显式 height */
}
```

### 2.2 点击无效 / placeholder 浮动但键盘不弹出

**现象**：设置 `line-height` 后，点击输入框 placeholder 文字上浮，但键盘不弹出，无法输入。

**根因**：微信小程序 `input` 是**原生组件**，在 WebView 上方浮动一个原生输入框覆盖层。`line-height` 会使 CSS 层的文字位置与原生覆盖层的实际触摸区域发生偏移，导致点击事件落在了原生层外面——CSS 层看到变化（文字上浮），但原生输入焦点未激活（键盘不弹）。

**正确做法**：
```css
.field-input-box {
  width: 100%;
  height: 88rpx;          /* 显式高度，撑开元素 */
  padding: 0 24rpx;       /* 只用水平 padding，垂直方向让原生 input 自行居中 */
  background: #F5F5F3;
  border: 2rpx solid var(--color-border);
  font-size: 28rpx;
  color: var(--color-text);
  box-sizing: border-box;
  /* 不要设置 line-height！ 原生 input 组件会处理垂直居中 */
}
```

**关键原则**：
- `height` → **必须设**，否则 padding + border-box 会裁剪内容
- `line-height` → **不能设**，会破坏原生 input 的触摸区域对齐
- `padding` → **只用水平方向** (`0 24rpx`)，垂直方向由原生组件自行居中

## 3. .env 文件在 Spring Boot 中不生效

**现象**：在项目目录放了 `.env` 文件，配置了 `COS_SECRET_ID` 等环境变量，但 `@ConfigurationProperties` 绑定的值为空，COS 初始化报 "COS 未配置"。

**根因**：Spring Boot 不原生支持 `.env` 文件。它只能从三个来源读取配置：
1. OS 系统环境变量
2. `application.yml` / `application.properties`
3. 命令行参数 `--key=value`

`.env` 是 Node.js / Docker Compose 生态的约定，Java 生态不认识。

**错误做法**：引入 `java-dotenv` 库在运行时加载 `.env`。有两个问题：
- `PropertySource` 方式注入时 key 的命名风格不匹配（`COS_SECRET_ID` vs Spring 的 relaxed binding `cos.secret-id`）
- `System.setProperty` 方式虽然能工作，但引入了非标准依赖，且 `.env` 文件里有 shell 语法（`$()`）需要额外处理

**正确做法（业界标准）**：Spring Boot 的 `${}` 占位符原生读取系统环境变量。

服务器上通过 systemd 的 `EnvironmentFile` 注入：
```ini
# /etc/systemd/system/moyu.service
[Service]
EnvironmentFile=/etc/moyu/config.env
ExecStart=/usr/bin/java -jar /moyu/moyu-server.jar
```

本地开发时在 IDE 中配置环境变量，或在终端 `export` 后再启动。

**为什么不用 .env**：
- `.env` 文件放在服务器上不够安全（容易误提交、权限管理不便）
- 业界 Java 项目标准是外部化配置 + 系统环境变量
- K8s/Docker 环境下直接用容器环境变量注入，也不依赖 .env
